Measurement Methodology
To compare implementation efficiency, two versions of the same algorithmic problem — a naive O(n²) approach and a single-pass O(n) approach — were compiled using the required strict flags (gcc -Wall -Werror -Wextra -pedantic -std=gnu89 -Wno-long-long) and executed on the same machine under the same conditions. Execution time was measured internally using clock() and CLOCKS_PER_SEC, which reports CPU time consumed by the process rather than wall-clock time affected by external interference. Each program was run three times to account for natural timing variability caused by OS scheduling, caching effects, and background system load. The three runs were then averaged to produce a more representative execution time than any single run could provide. A separate instrumentation exercise measuring a fixed-iteration loop (100,000,000 iterations) was also run three times to observe timing consistency for a workload with constant, predictable cost.

Observed Performance Differences
The results show a substantial gap between the two algorithms. Across three runs, the naive algorithm averaged 0.647626 seconds, with individual runs of 0.658340s, 0.632186s, and 0.652353s. The single-pass algorithm averaged 0.000026 seconds, with runs of 0.000027s, 0.000024s, and 0.000027s. Both algorithms produced correct and identical results (625025000 and 25000 respectively) across all runs, confirming that the comparison is fair and not an artifact of incorrect output. The naive implementation was measured to be approximately 24,909 times slower than the single-pass version. This large gap is consistent with the expected difference between O(n²) and O(n) growth: as input size increases, the naive algorithm's work grows quadratically while the single-pass algorithm's work grows linearly, producing a dramatic divergence in runtime even though both operate on the same input size. The separate iteration-count benchmark, which performed a fixed amount of work regardless of algorithm choice, showed comparatively stable results (0.101295s, 0.094249s, 0.092551s), reinforcing that timing variability between runs is normally small (roughly 8-9%) and that the 24,909x difference above cannot be explained by measurement noise alone.

Relation Between Runtime and Energy Consumption
Although this project does not measure watts directly, runtime serves as a reasonable proxy for CPU energy usage. A CPU executing instructions draws power for as long as it remains active; a program that runs longer keeps compute units, caches, and memory buses engaged for a longer duration, generally consuming more cumulative energy even if the instantaneous power draw is similar between programs. Given that the naive algorithm required roughly 0.65 seconds of CPU time versus 0.000026 seconds for the single-pass version, it is reasonable to infer that the naive implementation consumed substantially more energy to produce the same result. At scale — millions of executions across many machines — this difference compounds significantly, illustrating why algorithmic efficiency is a meaningful sustainability lever, not merely a performance one.

Limitations of the Experiment
This experiment has clear limitations. First, clock() measures CPU time, not actual power draw in watts, so no real energy or carbon figures can be derived — only a runtime-based proxy. Second, only three runs per program were collected; while consistent, this is a small sample and does not fully characterize variance under different system loads or hardware. Third, the measurements reflect one machine's CPU architecture, compiler version, and OS scheduler, so results are not necessarily generalizable. Finally, input size was fixed, so the results describe a single problem size rather than confirming scaling behavior across a range of inputs.

Practical Engineering Takeaway
Choosing an asymptotically efficient algorithm can produce runtime improvements of multiple orders of magnitude, as demonstrated by the ~24,909x difference measured here. Since runtime correlates with CPU activity and, by extension, energy consumption, algorithmic choice is a practical and measurable lever for writing more sustainable software — one that developers can evaluate using basic tools like clock() before reaching for specialized power-profiling equipment.
